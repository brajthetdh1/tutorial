What is Maven?

Maven is a powerful build automation and project management tool for Java projects, helping developers manage dependencies, automate builds, and standardize project structure.

Maven simplifies the process of building and managing Java projects. It allows you to:

  • Automate compilation, testing, and packaging of Java code
  • Manage external libraries and dependencies centrally
  • Generate project documentation and reports automatically
  • Enforce a standard project structure for all developers
  • Integrate easily with CI/CD pipelines and IDEs

By defining everything in a pom.xml file (Project Object Model), Maven ensures consistent builds across machines and environments.

Maven automates Java project management

<project xmlns="http://maven.apache.org/POM/4.0.0">
    <modelVersion>4.0.0</modelVersion>
    <groupId>com.example</groupId>
    <artifactId>my-app</artifactId>
    <version>1.0-SNAPSHOT</version>
</project>

This simple pom.xml declares a Java project with a group ID, artifact ID, and version. Maven uses it to manage the build lifecycle and dependencies automatically.

Understanding pom.xml

The pom.xml file is the heart of a Maven project. It defines project metadata, dependencies, plugins, and build configuration to automate and standardize your Java project builds.

Maven uses the pom.xml file (Project Object Model) to understand your project structure, required libraries, build instructions, and deployment configurations. It ensures that your builds are reproducible and consistent across environments.

Key Elements of pom.xml

  • modelVersion – Defines POM model version (usually 4.0.0).
  • groupId – Unique identifier for your project group or organization.
  • artifactId – Unique identifier for the project itself.
  • version – Project version (e.g., 1.0-SNAPSHOT).
  • dependencies – List of external libraries required by your project.
  • build – Build plugins, configuration, and resource directories.
  • repositories – Optional repositories to fetch dependencies from.

Example pom.xml

<project xmlns="http://maven.apache.org/POM/4.0.0"
         xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
         xsi:schemaLocation="http://maven.apache.org/POM/4.0.0
                             http://maven.apache.org/xsd/maven-4.0.0.xsd">
    <modelVersion>4.0.0</modelVersion>

    <groupId>com.example</groupId>
    <artifactId>my-app</artifactId>
    <version>1.0-SNAPSHOT</version>

    <dependencies>
        <dependency>
            <groupId>junit</groupId>
            <artifactId>junit</artifactId>
            <version>4.13.2</version>
            <scope>test</scope>
        </dependency>
    </dependencies>

    <build>
        <plugins>
            <plugin>
                <groupId>org.apache.maven.plugins</groupId>
                <artifactId>maven-compiler-plugin</artifactId>
                <version>3.10.1</version>
                <configuration>
                    <source>17</source>
                    <target>17</target>
                </configuration>
            </plugin>
        </plugins>
    </build>
</project>

This pom.xml declares a simple Java project with JUnit as a test dependency and the Maven Compiler Plugin configured for Java 17.

Comprehensive Guide to pom.xml Components

In Maven, the pom.xml (Project Object Model) file is the blueprint of your project. It defines project metadata, dependencies, plugins, build configuration, profiles, and repositories. Understanding each component ensures you can manage Java projects efficiently and consistently.

1. <modelVersion>

The <modelVersion> element specifies which POM model version your project is using. Currently, 4.0.0 is the standard. This allows Maven to parse and interpret the POM correctly.

2. <groupId>

<groupId> identifies your project group or organization uniquely. This is typically in reverse domain notation, e.g., com.example. It is used to prevent naming conflicts across projects and repositories.

3. <artifactId>

The <artifactId> specifies the unique name of your project. This name becomes the name of the resulting JAR, WAR, or other artifact. For example: my-app.

4. <version>

The <version> element defines the version of your project. Typical versions include:

  • 1.0-SNAPSHOT – development version
  • 1.0 – stable release version
  • 1.0.1 – patch or update version

Maven uses the version to manage artifact updates and dependencies consistently.

5. <packaging>

This optional element defines the package type your project produces. Default is jar. Other types include war, pom (for parent projects), and ear.

6. <properties>

The <properties> element allows you to define variables that can be reused across the POM. This improves maintainability and avoids hardcoding versions. 

<properties>
    <maven.compiler.source>17</maven.compiler.source>
    <maven.compiler.target>17</maven.compiler.target>
</properties>

7. <dependencies>

Defines external libraries required by your project. Maven automatically downloads these from repositories. Each dependency includes:

  • groupId – library group
  • artifactId – library name
  • version – specific library version
  • scope – compile, test, provided, or runtime
<dependency>
    <groupId>junit</groupId>
    <artifactId>junit</artifactId>
    <version>4.13.2</version>
    <scope>test</scope>
</dependency>

8. <dependencyManagement>

Allows parent POMs to define versions and scope for dependencies used by child modules. Useful for multi-module projects to enforce consistent versions.

9. <build>

Contains configuration for compiling, testing, packaging, and other build steps. You can define:

  • Plugins used in the build
  • Resource directories
  • Final artifact name
<build>
    <plugins>
        <plugin>
            <groupId>org.apache.maven.plugins</groupId>
            <artifactId>maven-compiler-plugin</artifactId>
            <version>3.10.1</version>
            <configuration>
                <source>17</source>
                <target>17</target>
            </configuration>
        </plugin>
    </plugins>
</build>

10. <profiles>

Profiles allow you to customize the build for different environments, e.g., development, testing, production. You can activate profiles via command line or automatically based on properties.

11. <repositories>

Optional repositories specify locations where Maven should search for dependencies. By default, Maven uses the central repository. 

<repositories>
    <repository>
        <id>company-repo</id>
        <url>https://repo.company.com/maven2</url>
    </repository>
</repositories>

Understanding Maven Lifecycle

Maven lifecycle defines the sequence of phases that take your project from source code to a deployable artifact. Each phase executes specific tasks automatically.

1. What is a Maven Lifecycle?

In Maven, a lifecycle is a predefined sequence of phases that execute tasks such as validation, compilation, testing, packaging, installation, and deployment. Maven has three built-in lifecycles:

  • default (build) lifecycle: handles project build, testing, packaging, and deployment.
  • clean lifecycle: handles cleaning the project directory (removing previous build artifacts).
  • site lifecycle: generates project documentation and reports.

2. Default Lifecycle Phases

The default lifecycle consists of several phases executed in order. Some of the key phases include:

  • validate: Validates the project structure and checks for required information.
  • compile: Compiles the source code of the project.
  • test: Runs unit tests using frameworks like JUnit.
  • package: Packages the compiled code into JAR/WAR/EAR artifacts.
  • verify: Performs any additional checks to verify the package is valid and meets quality criteria.
  • install: Installs the package into the local Maven repository for use as a dependency in other projects.
  • deploy: Copies the final package to the remote repository for sharing with other developers and projects.

Example command to execute a lifecycle phase: mvn clean install – which cleans, compiles, tests, packages, and installs the project in one go.

3. Clean Lifecycle

The clean lifecycle is used to remove artifacts created by previous builds. It has the following phases:

  • pre-clean: Executes tasks before cleaning.
  • clean: Deletes the target directory.
  • post-clean: Executes tasks after cleaning.

Example command: mvn clean

4. Site Lifecycle

The site lifecycle generates project documentation and reports. Phases include:

  • pre-site: Tasks before generating the site.
  • site: Generates the project site documentation.
  • post-site: Tasks after site generation.
  • site-deploy: Deploys the generated site to a web server.

Example command: mvn site

5. Maven Dependencies

Dependencies are external libraries your project requires. Maven downloads them automatically from repositories. 

<dependency>
    <groupId>junit</groupId>
    <artifactId>junit</artifactId>
    <version>4.13.2</version>
    <scope>test</scope>
</dependency>

Common scopes: compile, test, provided, runtime

6. Maven Plugins

Plugins extend Maven functionality, e.g., compilation, testing, packaging. 

<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-compiler-plugin</artifactId>
    <version>3.10.1</version>
    <configuration>
        <source>17</source>
        <target>17</target>
    </configuration>
</plugin>

7. Maven Repositories

  • Local Repository - on your machine: ~/.m2/repository
  • Central Repository - Maven Central, public
  • Remote Repository - corporate/private libraries
